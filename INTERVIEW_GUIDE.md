## 2. 项目介绍话术

### 2.1 1 分钟版本（电梯演讲）

> "我做的是一个基于 LangGraph 的**多 Agent 智能面试系统**。
>
> 核心是用一个 Supervisor 调度 5 个专用 Agent（简历匹配、面试出题、回答评分、报告生成、通用问答），
> 通过星型拓扑让它们协作完成模拟面试。
>
> 项目最大的工程挑战是**可靠性**：我设计了三层降级策略（Redis→SQLite→Memory），
> Agent 权限控制防止状态污染，混合路由（规则+LLM）把路由成本降了 80%。
> 即使 LLM 返回格式异常、Redis 挂了、某个 Agent 崩溃，系统都能降级而不中断。"

### 2.2 5 分钟版本（深度介绍）

```
1. 背景 (30s)
   "现在 AI 面试越来越普遍，但大多数只是简单的'问答对'。
   我想做的是真正的多轮、多维度、有评分的模拟面试系统。"

2. 架构设计 (90s)
   "我用 LangGraph 的 StateGraph 搭了星型拓扑：
    - Supervisor 是调度中心，负责所有路由决策
    - 5 个 Agent 各司其职，每个最多被连续调用 5 次防死循环
    - 状态通过 TypedDict 定义，权限通过装饰器白名单控制
    - 路由分三层：规则(80%，0 token) → LLM(20%) → 默认兜底"

3. 技术难点 (90s)
   "最大的难点是可靠性，因为 LLM 本身就是不稳定的：
    - LLM JSON 输出用了 4 层解析容错（直接parse→代码块提取→正则→默认值）
    - 每个 Agent 有独立的异常隔离，一个崩了不影响其他
    - 面试进行中不会被用户回答里的关键词误导路由
    - 三层持久化降级，Redis 挂了自动切 SQLite"

4. 成果 (60s)
   "最终实现了一个完整的面试闭环：上传简历→岗位匹配→5 道生成式题目→
   逐题提问→实时评分→综合报告。Web 端支持 SSE 流式输出，前端零框架纯 HTML。
   这个项目让我深刻理解了多 Agent 系统的设计权衡。"

5. 反思 (30s)
   "做得好的：权限控制和降级策略。可以改进的：引入 LangSmith 做 Agent 执行追踪，
   加入语音交互让体验更自然。"
```

## 3. 高频技术问题 & 参考答案

### 3.1 LangGraph 相关

#### Q: LangGraph 和 LangChain 的关系？为什么用 LangGraph？

```
LangChain 解决的是"单次 LLM 调用 + 工具"的问题，
LangGraph 解决的是"多次 LLM 调用如何编排"的问题。

LangGraph 的核心价值：
1. 显式的状态图 — 你能看到 Agent A → Agent B → Agent C 的流转路径
2. 内置的 Checkpoint — 每个 step 自动保存状态，支持断点续传
3. 循环支持 — 不像 DAG，LangGraph 支持 Agent 之间循环调用
4. 人机协作 — 可以在任意节点中断等待人工输入

我们的场景需要面试 Agent 和评分 Agent 交替执行 (循环)，
还要在出题后等待用户输入 (中断)，用 LangGraph 天然契合。
```

#### Q: StateGraph 中的状态是怎么管理的？

```
我们用 TypedDict 定义 MultiAgentState，包含约 25 个字段。
核心设计原则：
1. 字段分组：控制层 (Supervisor 写) / 业务层 (各 Agent 写) / 消息层 (共享)
2. _前缀私有：_agent_history 等追踪字段只有 Supervisor 能写
3. 渐进构建：状态不是一次性生成，而是随着 Agent 执行逐步填充
4. messages 用 add_messages reducer：追加而非替换

权限通过 AGENT_PERMISSIONS 字典 + with_permissions 装饰器实现。
装饰器拦截 Agent 返回值，过滤非白名单字段，捕获异常防止状态污染。
```

#### Q: Supervisor 路由是怎么决策的？

```
三层路由：

第 1 层：规则路由 (优先执行)
- 分析 state 的阶段信息 (有没有简历？有没有面试题？答了几题？)
- 检测用户意图 (关键词 + 正则)
- 优先级体系：面试保护 (最高) > 结束意图 > 简历 > 匹配 > 面试 > 聊天
- 覆盖 80% 场景，0 token 消耗

第 2 层：LLM 路由 (规则无法确定时)
- 将 state 摘要 + 对话上下文发给轻量模型 (deepseek-chat, temp=0)
- 最多 3 次重试，超时 30s
- 覆盖 20% 模糊场景

第 3 层：默认路由 (LLM 也失败时)
- 确定性兜底规则
- 按阶段顺序推进：简历→匹配→出题→答题→评分→报告
- 永不失败
```

### 3.2 RAG 相关

#### Q: 你的 RAG 系统是怎么设计的？

```
我们的 RAG 服务于三个场景：
1. 出题时：检索对应方向的知识库，生成贴近真实的面试题
2. 评分时：检索评分标准，让 LLM 的评分有据可依
3. 追问时：根据得分检索追问策略

技术实现：
- 文档分块：chunk_size=500, chunk_overlap=80 (经验值)
- 向量化：DashScope text-embedding-v2 (1536 维)
- 存储：ChromaDB 多集合 (每个知识文件一个 collection)
- 检索：top-k=3，同时检索多个 collection

关键经验：
- chunk_overlap 80 能有效避免关键信息在分块边界被切断
- 对于评分场景，检索评分标准比检索参考答案更重要
- 多 collection 比单 collection 混合检索效果更好（可以按方向精准检索）
```

#### Q: 遇到 RAG 检索不准确的情况怎么处理？

```
几个层次的应对：
1. 知识库质量：手动整理了面试知识库，确保内容质量
2. 检索策略：不同场景用不同的检索 query（出题用技能关键词，评分用题目文本）
3. 降级策略：检索不到结果时，LLM 用自己的知识出题（但仍然尽量结构化）
4. 评估反馈：评分时如果 RAG 标准不匹配，LLM 会根据通用标准打分

长期来看，可以加入用户反馈循环——用户对题目质量或评分准确性的反馈
可以用来调整检索策略或补充知识库。
```

### 3.3 容错与工程化

#### Q: 如果 LLM 返回的格式不对怎么办？

```
这是 LLM 应用中最常见的问题。我们用了多层容错：

JSON 解析的 4 层防护：
1. 直接 json.loads() — 理想情况
2. 正则提取 ```json...``` 代码块 — LLM 常见输出格式
3. 正则提取最外层 {...} — 乱码中提取 JSON
4. 返回安全的默认值 — 最终兜底，流程不中断

业务层的额外处理：
- 评分解析失败 → 默认给 5 分 + 友好提示
- 面试题解析失败 → 用正则逐题提取
- 永远不因为 LLM 格式问题让用户看到错误信息
```

#### Q: 怎么防止多 Agent 系统陷入死循环？

```
双重保护：

保护 1：连续调用检测
  同一非 Supervisor Agent 连续出现 5 次 → 强制 END
  另外有 pre-emptive check：路由前检查是否会变成第 6 次

保护 2：总量上限
  总步数 > 50 → 强制 END（Supervisor 的 Hard Cap 是 50）

这两个值是怎么确定的？
  5 次：resume_match 合法需要 4 次 (parse + match + show + select)
        多给 1 次 buffer 应对 re-match 场景
  50 次：正常面试约 15-25 步，50 步是正常值的 2-3 倍，
         绝对可以判定为异常

实际运行中从来没有误触发过。
```

#### Q: 三层降级中每一层倒下会有什么影响？

```
Redis 倒了：
  → 自动切 SQLite
  → 影响：性能略降（毫秒 → 微秒级影响，用户无感知）
  → 功能不受影响

Redis + SQLite 都倒了：
  → 切 Memory
  → 影响：服务重启后会话丢失（用户需要重新开始）
  → 但当前服务不中断

降级设计的哲学：宁可丢数据，不可停服务。
对于面试系统，用户需要的是面试不中断，
即使最坏情况回溯重启也比卡死好。
```

### 3.4 系统设计

#### Q: 如果面试过程中用户掉了，回来怎么继续？

```
这就是 LangGraph Checkpoint 机制发挥作用的地方：

1. 每个 step 后状态自动保存到 Checkpointer (Redis/SQLite/Memory)
2. thread_id 作为会话标识，绑定到状态快照
3. 用户恢复时：
   - 前端用保存的 session_id 发送请求
   - LangGraph 自动从上次中断的节点恢复
   - 用户看到的是"继续第 N 题"，而不是从头开始

具体实现：
  - ChatStore 保存对话历史 (前端展示用)
  - Checkpointer 保存 LangGraph 状态 (流程恢复用)
  - 两者通过 thread_id 关联
  - 前端通过 /api/sessions 可以列出历史会话
```

#### Q: 如果要支持 1000 人同时面试，你会怎么扩展？

```
评估瓶颈：
  - LLM API 调用：最慢（秒级），最贵
  - ChromaDB 向量检索：中（毫秒级）
  - FastAPI 服务：快（微秒级）

扩展方案：
1. LLM 层：
   - 多 API Key 负载均衡（DeepSeek + DashScope + Zhipu 互为备份）
   - 评分/出题异步化：用户回答后先回"正在评估..."，评分完再推送
   - ResponseCache 命中率提升（相似问题的评分可以复用缓存）

2. 向量层：
   - ChromaDB 读多写少，可以直接部署多副本
   - 知识库不常变，可以做 CDN 级别的分发

3. 服务层：
   - FastAPI 无状态 + Redis 做会话 → 天然可水平扩展
   - 加 Nginx 负载均衡
   - WebSocket/SSE 推送层独立

4. 排队机制：
   - 突发 1000 人同时调用 LLM → RateLimiter 排队
   - 非关键请求 (闲聊) 降低优先级
   - 关键请求 (评分) 优先处理
```

**运用本项目经验来回答**：

### 4.2 题目：如何设计一个 Agent 框架

```
可以从我们这个项目抽象出通用模式：

1. 核心抽象
   - Agent = (State, Process, Permissions)
   - State = TypedDict (共享状态)
   - Router = RuleEngine + LLMRouter + Fallback
   - Tool = 可被 Agent 调用的外部能力

2. 通信模式
   - 星型 (本项目): 中心化路由，简单可控
   - 网状: Agent 直接通信，灵活但复杂

3. 安全机制
   - 字段级权限 (本项目的 with_permissions)
   - 速率限制 (本项目的 RateLimiter)
   - 循环检测 (本项目的 check_and_enforce_loop_limits)
   - 预算控制 (token/时间/金钱)

4. 可观测性
   - 状态快照 (LangGraph Checkpoint)
   - 调用链追踪 (类似 LangSmith)
   - Agent 执行统计 (步数、耗时、错误率)
```

## 5. 代码深度问题

### 5.1 装饰器实现 Agent 权限控制

**问题**：解释 `with_permissions` 装饰器的工作原理，这样做的好处和代价是什么？

```python
# 参考答案

# 工作原理：
# 1. 装饰器接收 agent_name，从 AGENT_PERMISSIONS 查到该 Agent 的白名单
# 2. 执行原函数，获取返回的 dict
# 3. 遍历返回 dict 的每个 key，不在白名单中的过滤掉
# 4. 如果发生异常，捕获后返回错误信息，不污染状态

# 好处：
# - 零侵入：Agent 开发者不用关心权限，正常 return dict 就行
# - 声明式：权限配置集中在 AGENT_PERMISSIONS 字典，一目了然
# - 防御式：异常隔离、非法写入过滤，保护全局状态
# - 可观测：被拦截的写入有日志记录

# 代价：
# - 性能开销：每次返回都遍历 dict (可忽略，字段不超过 30 个)
# - 调试困惑：Agent 返回了某字段但没生效，开发者需要知道是权限拦截
# - 不够灵活：如果一个字段需要被两个 Agent 写，需要都加入白名单
```

### 5.2 为什么用 TypedDict 而不是 Pydantic Model？

```python
# LangGraph 要求 State 是可 JSON 序列化的 dict-like 对象
# TypedDict 本质就是 dict，LangGraph 完全支持

# Pydantic 的问题：
# - 序列化/反序列化有开销
# - 与 LangGraph Checkpoint 的兼容性问题
# - LangGraph 的 add_messages reducer 需要特殊处理

# TypedDict 的优势：
# - 纯 Python 类型注解，零运行时开销
# - LangGraph 原生支持
# - 简单直观，team 成员上手快
```

### 5.3 代码片段：路由的面试保护逻辑

**问题**：为什么面试进行中要把用户输入一律当成回答？会有什么边界情况？

```python
# 边界情况分析：

# 情况 1: 用户正常回答 → evaluate ✅
# 情况 2: 用户说"我不会这道题" → evaluate ✅ (评分 Agent 会处理)
# 情况 3: 用户说"我想换个岗位" → evaluate ✅ (回答完了 supervisor 再处理)
# 情况 4: 用户说"结束面试" → report ✅ (例外，严格匹配)

# 情况 3 的分析：
# 面试进行中用户说想换岗位，当下仍然走评分，
# 评分 Agent 会正常评分，然后 Supervisor 下次路由时检测到用户意图，
# 再转 resume_match。这样既不会打断当前题目的流程，又能在适当时机响应用户需求。
```

## 6. 行为面试

### 6.1 "你在这个项目中遇到的最大挑战是什么？"

```
建议回答方向：

技术挑战：LLM 的不确定性 vs 系统需要的确定性

"最大的挑战是让一个依赖 LLM 的系统保持可靠。
LLM 的输出是不可控的——JSON 格式可能不对、可能幻觉、可能超时。
面试系统又不能像聊天机器人一样'对不起我出错了'就完了。

我做的：
1. 每个 LLM 调用环节都设计了多层解析容错
2. Agent 间异常隔离，一个 Agent 崩了不影响全局状态
3. 路由不是全交给 LLM，而是规则优先
4. 每一层都有降级路径

收获：
深刻理解了'在一个不确定的组件（LLM）之上构建确定性系统'的工程方法。
这和在确定性的数据库上构建系统是完全不同的思维方式。"
```

### 6.2 "如果让你重新设计，你会改变什么？"

```
1. 引入可观测性
   "我会第一时间接入 LangSmith 或 LangFuse，现在排查 Agent 行为
   主要靠日志，如果有一个可视化的调用链追踪会高效很多。"

2. 测试策略
   "我会从项目开始就建立 Agent 的测试框架——用 mock LLM 响应来验证
   路由逻辑和各 Agent 的行为。目前的自检代码只在 __main__ 中。"

3. 配置外部化
   "系统提示词、路由规则、权限配置目前硬编码在代码中。
   我会把它们提取到配置文件，方便非开发人员调整。"

4. 评估体系
   "我会建立一个评估框架来量化面试质量——比如用户反馈、评分一致性、
   题目重复率等指标，用数据驱动迭代而非凭感觉。"
```

### 6.3 "你怎么保证 LLM 评分是公平的？"

```
这是一个很好的问题，因为它触及了 AI 系统的伦理维度。

当前的做法：
1. RAG 知识库提供了详细的评分标准（1-10 分每档的描述）
2. LLM 必须给出评分的具体理由，不能只给数字
3. 评分维度是固定的（技术、表达、深度、实践），防止随意打分
4. temperature 设为较低值，减少随机性

需要改进的：
1. 双盲评分：两个不同的 LLM 独立评分，差异太大时引入人工
2. 偏差检测：统计不同背景候选人的评分分布，检测系统性偏差
3. 可申诉：用户可以对评分提出异议，人工复核

最终观点：
LLM 评分不能替代人工面试，但作为一个练习工具，让候选人获得
即时反馈是有价值的。关键是让用户知道这是 AI 评分，仅供参考。
```
